# Heliolytics (Go Server) ‚Äî Coding Rules

This is the Go backend for the Heliolytics health analytics platform.
These rules apply to every file in this repo, for every session.

---

## Project Identity

- **Service name:** Heliolytics API
- **Language:** Go 1.22+
- **Deployment:** Oracle Cloud Instance (prod), local Docker (dev/staging)
- **Part of:** 3-repo system (App + Server + Web). Clients are Flutter app (`Heliolytics_App/`) and Next.js webapp (`Heliolytics_Web/`).

## The Single Most Important Rule

**One responsibility per file. No file should exceed ~150 lines.**

Handlers handle HTTP. Parsers parse bytes. Algorithms compute metrics.
If a file does more than one of these, split it.

---

## Naming

- Packages: short, lowercase, no underscores ‚Äî `api`, `store`, `domain`, `ble`
- Files: `snake_case.go`
- Types/interfaces: `PascalCase`
- Functions/methods: `camelCase` (private), `PascalCase` (exported)
- Constants: `PascalCase` for exported, `camelCase` for package-private
- Error variables: `Err` prefix ‚Äî e.g., `ErrNotFound`, `ErrUnauthorized`

---

## Code Style

- **Standard library first.** Only add a dependency if the stdlib can't do it cleanly.
- **Errors are values.** Return `(T, error)`. Never ignore errors.
- **No global state.** Dependencies are injected via constructors.
- **Short functions.** If a function doesn't fit on one screen, it's too long.
- **Table-driven tests.** Use `t.Run` with subtests.
- Run `gofmt`, `go vet`, and `golangci-lint` before every commit.

---

## API Design

- REST with JSON. Versioned: `/api/v1/...`
- Every response: `{ "data": ..., "error": null }` or `{ "data": null, "error": { "code": "...", "message": "..." } }`
- Auth: custom `X-Heliolytics-Token` header (HMAC-signed device token) + optional Firebase JWT for user-facing endpoints
- Rate limiting per device token. No anonymous access to data endpoints.
- HTTPS only in prod. Enforce `Strict-Transport-Security`.

---

## Security (non-negotiable)

- All API endpoints require a signed token. No exceptions.
- Token signing: HMAC-SHA256 with a server secret. Replay protection via nonce + timestamp (5-min window).
- No secrets in code. All secrets via environment variables.
- SQL: parameterized queries only (via sqlc). No string concatenation in queries.
- Logs: never log raw sensor data or tokens. Log correlation IDs only.

---

## Database

- **PostgreSQL + TimescaleDB** for all persistent storage.
- Migrations in `migrations/` ‚Äî numbered, never edited after applied.
- Queries generated by **sqlc** ‚Äî no hand-written SQL in Go code.
- Raw BLE bytes stored as BYTEA. Typed metrics in hypertables with timestamps.
- Schema: per-user data isolation via `user_id` foreign key on every table.

---

## Multi-tenancy

- Each user has a `user_id` (UUID). All data queries are scoped to `user_id`.
- Option A (SaaS): hosted Postgres ‚Äî all users share one DB, isolated by `user_id`.
- Option B (self-hosted): user runs their own Postgres instance; server is pointed at it via env var. Same codebase, different `DATABASE_URL`.
- The server doesn't care which mode it runs in ‚Äî it just reads `DATABASE_URL`.

---

## Testing

- Unit tests for: parsers, analytics algorithms, domain logic. No DB or HTTP.
- Integration tests for: API handlers (using `httptest`), store layer (using a test DB).
- No mocking the DB in unit tests ‚Äî use real queries against a test schema.

---

## Git

- Commits: `type(scope): message` ‚Äî e.g., `feat(api): add sync endpoint`
- Types: `feat`, `fix`, `chore`, `refactor`, `test`, `docs`
- One logical change per commit.
- No WIP commits on main.

---

## Deployment

- **Dev/staging:** `docker-compose up` spins up Postgres + TimescaleDB + server.
- **Prod:** Oracle Cloud Instance. Server binary in Docker. Managed Postgres or self-hosted TimescaleDB.
- Config via environment variables only. `.env.example` documents all required vars.
- Health check endpoint: `GET /health` ‚Äî returns 200 + DB connectivity status.
